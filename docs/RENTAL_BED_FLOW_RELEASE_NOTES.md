# ملاحظات إصدار نظام إيجار السراير

> **نظام كمبوند السبحي — الإصدار الحالي**
> آخر تحديث: يونيو 2026 | الأطوار: R24E → R24N

---

## 1. ملخص القرار

النظام **لم يعد يحجز الشقة بالكامل**. القرار الأساسي:

| القرار | التفاصيل |
|---|---|
| وحدة الحجز | **سرير** داخل شقة موحدة المواصفات |
| آلية الحجز | **واتساب** — لا يوجد دفع إلكتروني للإيجارات الحالية |
| Paymob / Payment | **معطل/محجوز** — لا يتم استدعاؤه في فلو الحجز الحالي |
| البيانات القديمة | سجلات الدفع التاريخية تبقى قراءة فقط ولا تؤثر على الفلو الحالي |

الشقق موحدة المواصفات (63 م²، غرفتان، حمام). الطالب يحجز **سريرًا** وليس الشقة كاملة. يبلغ الحد الأقصى للسراير **4 سراير لكل شقة**.

---

## 2. حالات السرير

| الحالة | المعنى | من يغيرها |
|---|---|---|
| `AVAILABLE` | متاح — يمكن للطالب حجزه | أدمن (من لوحة السراير)، أو تلقائي عند إلغاء حجز |
| `RESERVED` | محجوز مؤقتًا — تم تسجيل الطلب وانتظار مراجعة الأدمن | تلقائي عند إرسال طلب حجز |
| `RENTED` | مؤجر — الأدمن أغلق الاستفسار وأسند السرير رسميًا | أدمن فقط (إغلاق استفسار) |
| `OUT_OF_SERVICE` | خارج الخدمة — لا يُعرض ولا يُحجز | أدمن فقط |

> **قاعدة:** السرير `RENTED` **لا يرجع** `AVAILABLE` تلقائيًا. يتطلب إجراء أدمن صريح من لوحة السراير.

---

## 3. أولوية الحجز التلقائي

عند إرسال أي طلب حجز عام، يختار النظام **أقل رقم سرير متاح** تلقائيًا بدون تدخل الطالب:

| الطلب | السرير المحجوز | السراير المتاحة |
|---|---|---|
| الطلب الأول | سرير 1 | 3 |
| الطلب الثاني | سرير 2 | 2 |
| الطلب الثالث | سرير 3 | 1 |
| الطلب الرابع | سرير 4 | 0 → الإعلان يختفي |

**الآلية الفنية:** استعلام `SELECT ... FOR UPDATE SKIP LOCKED ORDER BY bed_number ASC LIMIT 1` داخل transaction لمنع الحجز المزدوج في نفس اللحظة.

لا يوجد اختيار يدوي لرقم السرير من الواجهة العامة حاليًا.

---

## 4. ظهور واختفاء الإعلان

```
عدد السراير المتاحة > 0  →  الإعلان ظاهر للعامة (status=ACTIVE, isPublished=true)
عدد السراير المتاحة = 0  →  الإعلان يختفي تلقائيًا (isPublished=false)
أدمن يرجع سرير → AVAILABLE  →  الإعلان يظهر مرة أخرى (isPublished=true)
```

تُحدَّث العدادات (`pendingBeds`, `rentedBeds`, `availableBeds`) تلقائيًا بعد كل عملية حجز أو إلغاء أو تغيير حالة سرير عبر `syncListingCountersFromBeds`.

## 5. فلو الطالب العام

```
الطالب يفتح الإعلان
    └── يرى: السراير المتاحة / إجمالي السراير
        └── يضغط "طلب تواصل وحجز سرير"
            └── يملأ: الاسم + رقم الموبايل + الرقم القومي
                └── يضغط "إنشاء طلب حجز سرير"
                    └── النظام يحجز أقل سرير متاح تلقائيًا
                        └── يُولَّد رسالة واتساب تتضمن:
                            ├── رقم السرير المحجوز: سرير N
                            └── عدد السراير المتاحة بعد الطلب: X
                                └── يضغط "إرسال طلب حجز السرير عبر الواتساب"
                                    └── يفتح جروب الواتساب + يلصق الرسالة
```

**لا يوجد دفع إلكتروني في أي خطوة من هذا الفلو.**

### محتوى رسالة الواتساب

```
طلب حجز سرير:
- الاسم بالكامل: [اسم الطالب]
- رقم الموبايل: [رقم الطالب]
- الرقم القومي: [رقم قومي الطالب]
- عنوان الإعلان: [عنوان الإعلان]
- معرف الإعلان (ID): [listing.id]
- معرف الشقة (Unit ID): [listing.unitId]
- رابط الإعلان العام: https://...
- إيجار الشقة الشهري: [السعر]
- مبلغ التأمين: [التأمين]
- حالة الشقة: [الحالة]
- السراير المتاحة: [العدد قبل الطلب]
- إجمالي السراير: [العدد الكلي]
- رقم السرير المحجوز: سرير N          ← يُضاف بعد تأكيد الخادم
- عدد السراير المتاحة بعد الطلب: X    ← يُضاف بعد تأكيد الخادم
```

> **لا توجد روابط أدمن داخل الرسالة.** لا يظهر `ownerPhone` ولا `ownerNationalId` للطالب.

## 6. فلو الأدمن

### لوحة السراير

في تفاصيل الإعلان داخل لوحة الأدمن، يجد الأدمن **لوحة السراير** التي تعرض:
- رقم كل سرير (سرير 1، سرير 2، ...)
- حالة السرير (badge ملون)
- الملاحظات (قابلة للتعديل)
- معرف الاستفسار / معرف الحجز المرتبط (إن وجد)

### إجراءات الأدمن على السرير

| الإجراء | الشرط | النتيجة |
|---|---|---|
| تحويل إلى `AVAILABLE` | من أي حالة | يرجع السرير متاحًا، يُزامن العدادات، قد يُعيد ظهور الإعلان |
| تحويل إلى `RESERVED` | من `AVAILABLE` فقط | يحجز السرير مؤقتًا يدويًا |
| تحويل إلى `RENTED` | من `AVAILABLE` أو `RESERVED` | يؤجر السرير رسميًا |
| تحويل إلى `OUT_OF_SERVICE` | من `AVAILABLE` أو `RESERVED` | يوقف السرير عن الخدمة |
| مزامنة العدادات | في أي وقت | يُعيد حساب العدادات من حالة السراير الفعلية |

### إغلاق الاستفسار (CLOSED)

لإغلاق استفسار وتحويله إلى حجز رسمي:
1. الأدمن يختار **حالة `CLOSED`** من قائمة الحالات.
2. تظهر قائمة اختيار السرير (السراير المتاحة أو المحجوزة بهذا الاستفسار فقط).
3. **لا يمكن الإغلاق بدون اختيار سرير** — النظام يرفض الطلب.
4. السرير المختار يصبح `RENTED`، تتزامن العدادات.

### إلغاء الاستفسار / الحجز (CANCELLED)

- **إلغاء استفسار:** يُحرر السراير في حالة `RESERVED` المرتبطة بهذا الاستفسار فقط → تعود `AVAILABLE`.
- **إلغاء حجز:** يُحرر السرير في حالة `RESERVED` المرتبط بهذا الحجز فقط → يعود `AVAILABLE`.
- **السراير في حالة `RENTED` لا تتأثر** بأي إلغاء استفسار أو حجز.

## 7. الخصوصية

| البيانات | متاح عامة | متاح للأدمن |
|---|---|---|
| `ownerPhone` | ❌ لا | ✅ نعم |
| `ownerNationalId` | ❌ لا | ✅ نعم |
| `pendingBeds` / `rentedBeds` (الأعداد الداخلية) | ❌ لا (محذوف من الاستجابة) | ✅ نعم |
| `availableBeds` | ✅ نعم (محسوب آمن) | ✅ نعم |
| `totalBeds` | ✅ نعم | ✅ نعم |
| روابط الأدمن | ❌ لا تظهر في رسائل الواتساب أو الواجهة العامة | — |

**القاعدة:** الاستجابات العامة تحذف `pendingBeds` و`rentedBeds` بشكل صريح عبر destructuring وتُبقي فقط `availableBeds` المحسوب.

## 8. المسارات الفنية المهمة

Base path: `/api/v1/rentals`

### المسارات العامة

| المسار | الوصف |
|---|---|
| `GET /listings` | قائمة الإعلانات العامة (ACTIVE + isPublished فقط) |
| `GET /listings/:slug` | تفاصيل إعلان عام |
| `POST /listings/:id/inquiries` | إنشاء استفسار + حجز سرير تلقائي (الفلو الرئيسي) |
| `POST /listings/:id/reservations` | حجز سرير مباشر بدون استفسار (لا يستدعي Paymob، يُعيد `bedNumber` و`remainingAvailableBeds`) |
| `GET /reservations/:id` | عرض حالة حجز (قراءة فقط، آمن) |
| `POST /listings/:id/contact-unlock` | **معطل** — يُعيد HTTP 410 فورًا |
| `GET /listings/:id/contact-access` | **قديم** — يُعيد `unlocked: false` دائمًا |
| `POST /payments/paymob/webhook` | للسجلات القديمة فقط — idempotent |

### مسارات الأدمن (تتطلب `Authorization: Bearer ADMIN_TOKEN`)

| المسار | الوصف |
|---|---|
| `GET /admin/listings/:listingId/beds` | قائمة سراير الإعلان مرتبة حسب رقم السرير |
| `PATCH /admin/beds/:bedId/status` | تغيير حالة سرير + مزامنة عدادات + تحديث isPublished |
| `POST /admin/listings/:listingId/beds/sync-counters` | مزامنة قسرية للعدادات من حالة السراير |
| `GET /admin/inquiries` | قائمة الاستفسارات |
| `GET /admin/inquiries/:id` | تفاصيل استفسار |
| `PATCH /admin/inquiries/:id/status` | تحديث حالة استفسار — `CLOSED` يتطلب `bedId` |
| `PATCH /admin/reservations/:id/confirm` | تأكيد حجز (يغير الإعلان إلى `RENTED`) |
| `PATCH /admin/reservations/:id/cancel` | إلغاء حجز + تحرير السرير `RESERVED` + مزامنة |

### مثال — طلب حجز سرير عام

```http
POST /api/v1/rentals/listings/:id/reservations
Content-Type: application/json

{
  "tenantName": "محمد أحمد",
  "tenantPhone": "+201012345678",
  "tenantEmail": "optional@email.com"
}
```

استجابة ناجحة (201):

```json
{
  "success": true,
  "data": {
    "reservation": { "id": "...", "status": "RESERVED", ... },
    "payment": null,
    "paymentUrl": null,
    "bedNumber": 1,
    "remainingAvailableBeds": 3
  }
}
```

خطأ عند عدم وجود سراير متاحة (409):

```json
{
  "success": false,
  "message": "لا توجد سراير متاحة لهذا الإعلان"
}
```

### مثال — إغلاق استفسار مع تحديد سرير

```http
PATCH /api/v1/rentals/admin/inquiries/:id/status
Authorization: Bearer ADMIN_TOKEN
Content-Type: application/json

{
  "status": "CLOSED",
  "bedId": "bed-uuid-here"
}
```

## 9. حالة Paymob / Payment

| الجانب | الحالة |
|---|---|
| استدعاء Paymob في فلو حجز السرير | ❌ معطل تمامًا |
| إنشاء `RentalPayment` في فلو حجز السرير | ❌ لا يتم |
| `contact-unlock` payment | ❌ معطل — HTTP 410 |
| البيئة (`PAYMOB_API_KEY`, ...) | محفوظة في `.env` لأغراض المراجعة التاريخية فقط |
| سجلات الدفع القديمة في قاعدة البيانات | ✅ تبقى قراءة فقط |
| واجهة الأدمن — تاريخ الدفع | يظهر مع تسميات تاريخية: "طلب قديم" / "قفل قديم" |

> **قاعدة صارمة:** لا تُفعّل مسارات بدء الدفع (`startReservationPayment`/`startContactUnlockPayment`) في فلو الإيجار الحالي. كلا المسارين محجوزان أو يُعيدان خطأ فوريًا.

## 10. قائمة اختبار دخان يدوي

يجب تنفيذ هذه الخطوات على **إعلان تجريبي آمن** (وليس إعلانًا حقيقيًا للسكان) قبل إعلان الاستعداد الكامل للإنتاج.

- [ ] **1.** إنشاء أو تحديد إعلان تجريبي بـ 4 سراير كلها `AVAILABLE`، حالة `ACTIVE`، `isPublished=true`.
- [ ] **2.** إرسال طلب حجز من الواجهة العامة.
- [ ] **3.** التأكد من أن **سرير 1** أصبح `RESERVED` وعدد السراير المتاحة = **3**.
- [ ] **4.** إرسال طلب حجز ثانٍ.
- [ ] **5.** التأكد من أن **سرير 2** أصبح `RESERVED` وعدد السراير المتاحة = **2**.
- [ ] **6.** إرسال طلب حجز ثالث.
- [ ] **7.** التأكد من أن **سرير 3** أصبح `RESERVED` وعدد السراير المتاحة = **1**.
- [ ] **8.** إرسال طلب حجز رابع.
- [ ] **9.** التأكد من أن **سرير 4** أصبح `RESERVED` وعدد السراير المتاحة = **0** والإعلان **اختفى من الواجهة العامة**.
- [ ] **10.** من الأدمن، تحويل **سرير 4** إلى `AVAILABLE`.
- [ ] **11.** التأكد من أن الإعلان **ظهر مرة أخرى** في الواجهة العامة مع **سرير 1 متاح**.
- [ ] **12.** من الأدمن، إغلاق استفسار واختيار سرير محدد.
- [ ] **13.** التأكد من أن السرير المختار أصبح `RENTED`.
- [ ] **14.** إلغاء استفسار آخر يملك سريرًا `RESERVED`.
- [ ] **15.** التأكد من أن السرير `RESERVED` **رجع** `AVAILABLE`.
- [ ] **16.** التأكد من أن السراير `RENTED` **لم تتأثر** بالإلغاء.

## 11. القيود المعروفة

| القيد | التفاصيل |
|---|---|
| لا يوجد دفع إلكتروني | الحجز الحالي عبر واتساب فقط |
| لا يوجد اختيار يدوي لرقم السرير | النظام يختار تلقائيًا حسب الأولوية |
| سجلات دفع تاريخية | قد تظهر في الأدمن كبيانات قراءة فقط — لا تؤثر على الفلو الحالي |
| الاختبار الكامل | يتطلب تنفيذ قائمة اختبار الدخان اليدوي على إعلان تجريبي |
| مدة الحجز المؤقت | لا توجد انتهاء صلاحية تلقائية حاليًا للحالة `RESERVED` — يتطلب إجراء أدمن |
